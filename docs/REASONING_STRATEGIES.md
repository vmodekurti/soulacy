# Pluggable Reasoning Strategies (Story E15)

Soulacy's reasoning loop dispatches by strategy **name** through the SDK
factory registry. The built-ins (`react`, `plan_execute`) register exactly
like any extension would; custom loops (Tree of Thought, Self-Reflection,
Consensus Swarms, …) plug in without touching host code.

## The contract

Everything an author needs lives in `sdk/reasoning` (stdlib-only, frozen per
SDK major version — see `sdk/README.md` for the compatibility policy):

```go
type Strategy interface {
    Run(ctx context.Context, env Env, taskInput string) ([]Step, ReflectResponse)
}

type Env struct {
    Config Config       // limits + system prompt + tool names from SOUL.yaml
    LLM    LLMBackend   // Think / Plan / Reflect
    Tools  ToolExecutor // dispatches ToolCalls
}
```

Rules a conforming strategy MUST follow:

1. **Always return a usable `ReflectResponse`.** Tool failures become
   observations (`Observation.Error` → `"tool error: …"` content), never
   aborts. Degraded output beats no output.
2. **Respect the limits** in `env.Config`: `MaxSteps`, `StepTimeout` (per
   step), and the run context (the host wraps `TotalTimeout` around `Run`).
3. **Return the step trace.** Every think-act-observe cycle should appear in
   the returned `[]Step` — observability surfaces (Activity, run metrics,
   workboard history) render it.
4. **No subprocesses, no goroutine leaks.** Strategies are pure loop logic;
   the host owns processes and lifecycle.

## Registering

```go
import (
    "github.com/soulacy/soulacy/sdk/reasoning"
    "github.com/soulacy/soulacy/sdk/registry"
)

func init() {
    registry.MustRegisterReasoningStrategy("tree_of_thought",
        func(cfg map[string]any) (reasoning.Strategy, error) {
            return &treeOfThought{}, nil
        })
}
```

The factory's `cfg` map is reserved for strategy-specific tuning (currently
the host passes nil); per-run inputs arrive via `Env`.

Compiled-in strategies are linked through the generated blank-import file
(`cmd/soulacy/builtins_gen.go`, regenerated by `go generate ./...` from
`scripts/genbuiltins`). A custom distribution adds its strategy package to
that list — the same mechanism the E12 flavored-binary tool automates.

## Selecting from SOUL.yaml

```yaml
reasoning:
  strategy: tree_of_thought   # any registered name; "auto" picks react/plan_execute
  max_steps: 8
  step_timeout: 30s
  total_timeout: 180s
```

`LoopConfigFromDefinition` carries the name through unchanged;
`Loop.Run` resolves it via `registry.NewReasoningStrategy`.

## Fallback semantics

- `auto` (or empty) → keyword heuristic picks `plan_execute` or `react`.
- Unknown name, factory error, or nil strategy → **ReAct fallback**. A typo'd
  SOUL.yaml never bricks an agent; the run completes with the default loop.

## Conformance

`internal/reasoning/custom_strategy_test.go` is the seed of the E11 kit: it
registers a custom strategy, selects it through the SOUL.yaml path, and
asserts it runs end-to-end plus the fallback behaviour. Strategy authors
should replicate those assertions against their own implementation.
